What Is Hermes WebUI? #

Hermes WebUI is a lightweight, dark-themed web application that provides a browser-based interface to the Hermes Agent. It delivers full 1:1 parity with the CLI experience - everything you can do from a terminal, you can do from this UI. Built with Python and vanilla JavaScript, it requires no build step, no framework, and no bundler.

Three-Panel Layout

Left sidebar for sessions, center for chat, right for workspace file browsing. Modeled on the Claude-style interface.

Zero Extra Setup

Uses your existing Hermes Agent and configured models. No additional services needed.

Mobile-First

Responsive SPA designed for phone browsers. Hamburger nav, touch-friendly targets, full-height chat.

Docker Ready

Pre-built multi-arch images (amd64 + arm64) on GHCR. Single, two, or three-container setups.

Secure by Default

Optional password auth, signed HMAC cookies, SRI-pinned CDN resources, security headers.

Internationalized

8 locales: English, Chinese (Simplified and Traditional), Korean, Russian, Spanish, German, Portuguese.

Key Principle: Hermes WebUI is not a replacement for Hermes Agent - it is a surface for it. The WebUI reads your existing ~/.hermes directory, shares the same config, sessions, skills, and memory. Nothing is duplicated or proxied.

Architecture #

Hermes WebUI is a single-process Python application that imports Hermes Agent modules directly - there is no HTTP communication between separate services.

Quick Start #

Get Hermes WebUI running in under 2 minutes.

  1. Clone the repository
    git clone https://github.com/nesquena/hermes-webui.git hermes-webui
cd hermes-webui
  2. Run the start script
    ./start.sh

    Installs dependencies, checks for Hermes Agent, starts server on http://localhost:8787.

  3. Open in browser

    Navigate to http://localhost:8787. The onboarding wizard guides you through provider setup on first run.

That is it. No build step, no bundler, no framework. If you are on a VPS, see Docker and Remote Access below.

Installation Methods #

Local Installation via start.sh

git clone https://github.com/nesquena/hermes-webui.git hermes-webui
cd hermes-webui
./start.sh

The script will:

  • Check for Python 3.10+ and install dependencies via pip
  • Verify Hermes Agent is installed and accessible
  • Create ~/.hermes/webui-mvp state directory if needed
  • Start the server on 127.0.0.1:8787
  • If running over SSH, print the SSH tunnel command automatically
Prerequisites: Python 3.10+, Hermes Agent installed, API keys configured in ~/.hermes/.env or ~/.hermes/config.yaml.

Docker Installation (Recommended for VPS)

docker pull ghcr.io/nesquena/hermes-webui:latest
docker run -d \
  -e WANTED_UID=$(id -u) -e WANTED_GID=$(id -g) \
  -v ~/.hermes:/home/hermeswebui/.hermes \
  -v ~/workspace:/workspace \
  -p 127.0.0.1:8787:8787 \
  ghcr.io/nesquena/hermes-webui:latest

See Docker Setup for full details.

Manual Setup

git clone https://github.com/nesquena/hermes-webui.git
cd hermes-webui
mkdir -p ~/.hermes/webui-mvp
pip install -r requirements.txt
export HERMES_WEBUI_HOST=0.0.0.0
export HERMES_WEBUI_PORT=8787
python -m uvicorn api.main:app --host 0.0.0.0 --port 8787

Docker Setup #

Docker is the recommended deployment method for VPS. Three compose configurations:

Single-Container

Recommended. WebUI runs agent in-process. Simplest setup. Uses docker-compose.yml

Two-Container

Agent and WebUI separate. Isolation between gateway and chat UI. Uses docker-compose.two-container.yml

Three-Container

Agent + Dashboard + WebUI. Full monitoring stack. Uses docker-compose.three-container.yml

Single-Container Setup

docker-compose.yml
version: '3.8'
services:
  hermes-webui:
    image: ghcr.io/nesquena/hermes-webui:latest
    container_name: hermes-webui
    ports:
      - "127.0.0.1:8787:8787"
    volumes:
      - hermes-home:/home/hermeswebui/.hermes
      - ~/workspace:/workspace
    environment:
      - WANTED_UID=${UID:-1000}
      - WANTED_GID=${GID:-1000}
      - HERMES_WEBUI_STATE_DIR=/home/hermeswebui/.hermes/webui-mvp
    restart: unless-stopped
volumes:
  hermes-home:

Two-Container Setup

Use when you run Hermes Agent separately (for Telegram/Discord/CLI) and want WebUI as an additional surface.

docker-compose.two-container.yml
version: '3.8'
services:
  hermes-agent:
    image: nousresearch/hermes-agent:latest
    container_name: hermes-agent
    volumes:
      - hermes-home:/root/.hermes
      - hermes-agent-src:/opt/hermes
    environment:
      - HERMES_HOME=/root/.hermes
    restart: unless-stopped
  hermes-webui:
    image: ghcr.io/nesquena/hermes-webui:latest
    container_name: hermes-webui
    depends_on:
      - hermes-agent
    ports:
      - "127.0.0.1:8787:8787"
    volumes:
      - hermes-home:/home/hermeswebui/.hermes
      - hermes-agent-src:/home/hermeswebui/.hermes/hermes-agent
      - ~/workspace:/workspace
    environment:
      - HERMES_WEBUI_HOST=0.0.0.0
      - HERMES_WEBUI_PORT=8787
      - HERMES_WEBUI_STATE_DIR=/home/hermeswebui/.hermes/webui-mvp
      - WANTED_UID=${UID:-1000}
      - WANTED_GID=${GID:-1000}
    restart: unless-stopped
volumes:
  hermes-home:
  hermes-agent-src:
Named volumes are critical: The two-container setup uses named Docker volumes by default. Named volumes solve the UID/GID problem automatically. Do not replace them with bind mounts unless you understand the permission implications.

UID/GID Configuration

The container auto-detects your host user UID/GID. Set explicitly if needed:

id -u  # e.g., 1000
id -g  # e.g., 1000
# Set in .env: UID=1000, GID=1000

Bind-Mount Migration (Advanced)

volumes:
  hermes-home:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/youruser/.hermes

Requirements: Host directory must be readable by container UID. All containers sharing the volume must run as the SAME UID/GID. If host .env is mode 0640, set HERMES_SKIP_CHMOD=1.

Dependency Installation

At startup, the init script runs:

uv pip install /home/hermeswebui/.hermes/hermes-agent

This installs Hermes Agent and all dependencies into the WebUI Python environment on first boot. Takes 1-2 minutes.

Remote Access #

SSH Tunnel (Recommended)

Safest method. No ports exposed to the internet.

ssh -N -L 8787:127.0.0.1:8787 user@your-server.com

Then open http://localhost:8787. The start.sh script prints this command automatically when it detects SSH.

Tailscale / ZeroTier

  1. Install Tailscale on VPS
    curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
  2. Install Tailscale on local machine
    curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
  3. Access via Tailscale IP
    http://<tailscale-ip>:8787

Direct Network Exposure

Security warning: Only expose directly if you set HERMES_WEBUI_PASSWORD. Without a password, anyone who can reach the port has full access to your agent.
# In docker-compose.yml:
ports:
  - "8787:8787"
environment:
  - HERMES_WEBUI_PASSWORD=your-strong-password

Reverse Proxy (Caddy)

hermes-webui.yourdomain.com {
    reverse_proxy localhost:8787
}

Caddy auto-provisions Let's Encrypt certificates.

Onboarding Behind Reverse Proxy

If behind nginx/Caddy/Traefik or SSH tunnel, the onboarding IP check may block you. Fixed in v0.50.22+. For older versions:

# Option 1: Set env var
HERMES_WEBUI_ONBOARDING_OPEN=1
# Option 2: Set password before first run
HERMES_WEBUI_PASSWORD=your-password
# Option 3: Upgrade to v0.50.22+

Chat and Streaming #

The chat interface is the core of Hermes WebUI. Every CLI workflow has a web equivalent.

Sending Messages

Type in the composer and press Enter. Messages are sent via SSE (Server-Sent Events) for real-time streaming.

Streaming Behavior

  • rAF-throttled rendering: Token rendering throttled via requestAnimationFrame to prevent UI jank
  • SSE auto-reconnect: If connection drops (common with SSH tunnels), stream automatically reconnects
  • Send while streaming: Queue next message while agent responds - sent inline after current response

Model Selection

The composer footer includes a model picker showing all providers from your config.yaml:

  • Multi-provider: OpenAI, Anthropic, Google, OpenRouter, xAI, GLM, DeepSeek, Mistral, MiniMax, Kimi, Nous Portal, custom OpenAI-compatible
  • Dynamic discovery: Ollama, LM Studio, vLLM models discovered live via /v1/models
  • Free-form OpenRouter: Type any model name with autocomplete
  • Per-conversation: Each session can use a different model
  • LM Studio first-class: Dedicated provider with live discovery even without env vars injected

Reasoning Display

  • Claude thinking blocks: Rendered inline with expand/collapse
  • o3 reasoning blocks: Distinct styling
  • Reasoning effort selector: low/medium/high + /reasoning command

Rich Content Rendering

  • Markdown: Full CommonMark with GFM extensions
  • Code blocks: Prism.js syntax highlighting (Python, JS, bash, JSON, SQL, etc.). Copy button with "Copied!" feedback
  • Mermaid diagrams: Rendered inline from ```mermaid blocks
  • KaTeX math: Inline $...$ and block $$...$$ rendering

Cancel and Retry

  • Stop button: Cancel streaming at any time
  • Retry: Delete last assistant message and resend with one click

Context Usage Indicator

Circular ring badge in composer footer shows input tokens, output tokens, estimated cost, and context fill percentage. Toggleable in Settings or via /usage.

File Attachments

  • Drag-and-drop or click to attach
  • Persist across page reloads
  • Images displayed inline

Skills and Memory #

Skills System

Hermes Agent has a self-improving skills system that auto-generates reusable skills from complex tasks. WebUI provides full CRUD:

  • Browse: View all skills in Control Center
  • Create/Edit/Delete: Full skill management

Skills stored as markdown in ~/.hermes/skills/, directly editable.

Memory System

Hermes retains context across sessions via:

  • User profile: Persistent preferences and identity
  • Agent notes: Manually written or auto-generated notes
  • Self-improving skills: Skills that evolve from experience

Memory files stored in ~/.hermes/memory/ as editable markdown. WebUI provides memory write surface in Control Center.

Memory inspectability: Unlike cloud agents, Hermes memory is plain markdown you can read, edit, and audit. You own your memory data.

Cron and Scheduled Jobs #

Self-hosted scheduling that fires while you are offline and delivers results to multiple platforms.

Creating Cron Jobs

  • CRUD: Create, view, switch, edit, delete from Control Center
  • Cron expressions: Standard cron syntax
  • Delivery targets: Telegram, Discord, Slack, Signal, email, and more

How It Works

Cron jobs managed by Hermes Agent scheduler. Execution happens in the agent process. Jobs fire even when WebUI is closed, as long as agent is running.

Use Cases

  • Daily data aggregation and report generation
  • Weekly file organization and cleanup
  • Scheduled reminders and notifications
  • Periodic web scraping and data collection

Voice Mode #

Turn-based voice interaction using browser Web Speech API.

  1. Activate: Click voice button in composer
  2. Speak: Browser captures audio via Web Speech API (STT)
  3. Send: Transcribed text sent as message
  4. Listen: Agent response read aloud via TTS
  5. Loop: Auto-resumes listening after response completes

Features

  • Progressive enhancement: Hidden if Web Speech API unsupported
  • Pulsing indicator: Visual feedback when listening
  • TTS settings: Respects existing voice/rate/pitch config
  • Error handling: Bail-out on not-allowed and audio-capture errors with toast
Browser support: Requires Chrome, Edge, or Safari. Firefox does not support SpeechRecognition. Hidden automatically in unsupported browsers.

Security and Authentication #

Password Authentication

HERMES_WEBUI_PASSWORD=your-secret-password

Uses signed HMAC HTTP-only cookies. Cookie not accessible via JavaScript, preventing XSS session hijacking.

Rate Limiting

Login rate limiting persists across restarts. Failed attempts stored in STATE_DIR/.login_attempts with atomic writes and 0600 permissions. Buckets prune on load.

Security Headers

All responses include security headers. CDN resources SRI-pinned to prevent supply chain attacks.

API Redaction

API keys redacted by default. Self-hosted users can disable via Settings toggle. Lazy check at call-time, default ON for safety.

Workspace Allowlist

Paths validated against workspaces.json. Directory traversal prevented by _resolve_workspace.

Offline Recovery

When browser cannot reach Hermes, high-contrast offline banner appears. App probes /health automatically and refreshes on reconnection. Stream errors deferred while offline.

PWA Security

Service-worker shell assets use network-first strategy (with cache fallback). PWA install + offline launch preserved via install-time pre-caching.

Troubleshooting #

Onboarding blocked behind reverse proxy

Fix: Upgrade to v0.50.22+, or set HERMES_WEBUI_ONBOARDING_OPEN=1, or set HERMES_WEBUI_PASSWORD before first run.

Container permission issues

Fix: Set WANTED_UID and WANTED_GID to match host user. Use id -u and id -g.

Agent dependencies not installed

Fix: Check hermes-agent-src volume is mounted correctly. Check container logs for uv pip install output.

SSH tunnel connection drops

Fix: SSE auto-reconnect handles this. Add ServerAliveInterval 60 to SSH config if persistent.

Docker port binding - cannot access from other machines

Fix: Change port to "8787:8787" in docker-compose.yml and set HERMES_WEBUI_PASSWORD.

Two-container agent resolution

Fix: Use named volumes from docker-compose.two-container.yml. Do not replace with bind mounts.

Podman compatibility

Fix: Upgrade to Podman 4+, or use single-container setup, or use community image sunnysktsang/hermes-suite.

Workspace not in allowlist

Fix: Ensure workspace path is in workspaces.json.

Offline banner will not disappear

Fix: Check network connection and that Hermes server is running. Banner appears when navigator.onLine === false or /health probe fails.

Stream errors on page load

Fix: Normal if previous session was streaming when page closed. Click different session and back, or refresh.
